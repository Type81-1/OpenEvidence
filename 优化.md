# 赛道一：临床证据助手优化计划

## 当前状态

项目已经跑通基础 MVP：

```text
中文医学问题
-> PubMed 检索
-> 文献写入 Chroma 向量数据库
-> RAG 检索相关证据
-> 调用学校大模型生成中文回答
```

当前项目适合继续作为「赛道一：临床证据助手」推进。后续优化重点不是继续堆模型，而是提高证据可信度、引用可追溯性、演示体验和评估完整性。

## 优化目标

1. 让系统回答中的每条关键结论都能追溯到真实证据。
1. 支持用户输入不同临床问题，而不是只跑固定样例。
1. 接入更多证据来源，包括 PubMed 文献、ClinicalTrials.gov 临床试验和指南/共识资料。
1. 形成可展示的前端页面或清晰命令行演示流程。
1. 准备固定测试问题和对比评估结果，支撑项目报告。

## P1：优化 PubMed 结构化解析和 PMID 引用

状态：已完成。

### 当前问题

现在系统把多篇 PubMed XML 合在一起写入 Chroma，并且只用第一个 PMID 作为文档 ID。这样最终回答里的 `[1]`、`[2]` 不够严谨，无法清楚对应真实文献。

### 要做什么

- 将 PubMed XML 解析成结构化字段：

```text
PMID
标题
摘要
期刊
年份
DOI
PubMed URL
```

- 每篇文献单独入库。
- 每个文本块保存 metadata：

```text
source_type = pubmed
pmid
title
journal
year
url
chunk_index
```

- 回答引用从普通 `[1]` 升级为：

```text
[PMID: 42468089]
```

或：

```text
[1] PMID: 42468089
```

### 验收标准

- 最终回答的每个关键结论都能对应真实 PMID。
- 回答末尾能列出参考证据：

```text
[1] PMID: xxxx | 标题 | 期刊 | 年份 | URL
```

## P2：支持用户输入问题

状态：进阶版已完成。

### 当前问题

原来 `main.py` 里的问题和 PubMed 检索词是写死的：

```python
question_cn = "体检发现血脂偏高..."
pubmed_query_en = "dyslipidemia lifestyle intervention..."
```

现在已经改成支持交互式输入、命令行参数输入，以及大模型自动生成英文 PubMed query。

### 要做什么

- 交互式输入：

```bash
python main.py
```

- 命令行直接传入问题：

```bash
python main.py --question "高血压患者为什么有时要长期吃药？有哪些指南或研究依据？"
```

- 只生成英文 PubMed query，不检索文献：

```bash
python main.py \
  --question "体检发现血脂偏高，生活方式干预和药物治疗分别有哪些证据？" \
  --show-query-only
```

- 手动覆盖 PubMed query：

```bash
python main.py \
  --question "体检发现血脂偏高，生活方式干预和药物治疗分别有哪些证据？" \
  --pubmed-query "dyslipidemia lifestyle intervention statin therapy cardiovascular risk"
```

- query 生成模式：

```text
auto：优先学校大模型生成，失败后用本地保底规则
llm：必须使用学校大模型生成
fallback：只使用本地保底规则
```

### 验收标准

- 不改代码也能测试不同问题。
- 至少支持 5 个固定演示问题。
- 可以调用学校大模型把中文问题改写成英文 PubMed query。
- 大模型不可用时仍可用本地规则生成保底 query。

## P3：接入 ClinicalTrials.gov 临床试验证据

状态：已完成。

### 当前问题

原来项目里有 `clinical_trials.py`，但没有接入 `main.py` 主流程。现在已经改成：

```text
PubMed 检索和入库
-> ClinicalTrials.gov 检索和入库
-> RAG 同时检索 PubMed 文献证据和临床试验证据
-> 回答中可引用 PMID 和 NCT ID
```

### 要做什么

- 根据同一个英文检索词查询 ClinicalTrials.gov。
- 提取临床试验结构化字段：

```text
NCT ID
试验标题
研究状态
简要描述
干预措施
疾病条件
招募状态
地点
ClinicalTrials.gov URL
```

当前额外提取：

```text
研究类型
试验分期
样本量
开始日期
完成日期
申办方
```

- 写入 Chroma，并在 metadata 中标记：

```text
source_type = clinical_trial
nct_id
title
status
url
```

- 最终回答中单独增加：

```text
## 临床试验证据
```

- 运行参数：

```bash
python main.py --max-trials 3
```

- 如需临时关闭 ClinicalTrials.gov：

```bash
python main.py --no-clinical-trials
```

- `.env` 可配置：

```bash
CLINICAL_TRIALS_ENABLED=true
```

### 验收标准

- 回答中可以引用 NCT ID。
- 系统能区分 PubMed 文献证据和临床试验证据。
- ClinicalTrials.gov API 失败时不阻断 PubMed 主流程。

## P4：加入指南或共识资料

## P5：优化 Prompt 和安全边界

状态：已完成。

### 当前问题

当前 prompt 已经升级为更严格的临床证据助手 prompt，可以区分 PubMed、ClinicalTrials.gov 和指南/共识证据。

### 要做什么

- 在 prompt 中明确要求：

```text
1. 区分证据类型：PubMed 文献、ClinicalTrials.gov 临床试验登记、指南/共识。
2. 每个关键结论必须引用 PMID、NCT ID 或指南 ID 对应的证据编号。
3. 证据不足时必须明确说明。
4. 不输出个人诊断、处方剂量或停药建议。
5. 明确提醒需要医生结合具体情况判断。
6. 不编造 PMID、NCT ID、指南 ID、统计数字或治疗阈值。
7. 指南/共识只能说明推荐方向，不能替代个体化诊疗。
```

- 回答结构建议改为：

```text
## 简短回答
## 主要证据
## 生活方式干预
## 药物治疗
## 临床试验证据
## 指南/共识证据
## 证据局限
## 临床意义
## 参考证据
```

### 验收标准

- 回答更稳定、更结构化。
- 遇到证据不足时不会强行给结论。

## P7：做测试问题和对比评估

状态：已完成。

### 当前问题

`evaluation.py` 原来只检查是否有引用、是否有“不足”、回答长度。现在已升级为固定测试集 + 纯大模型/RAG 对比评估脚本。

### 要做什么

准备 5 个固定问题：

```text
1. 高血压患者为什么有时要长期吃药？有哪些指南或研究依据？
2. 体检发现血脂偏高，生活方式干预和药物治疗分别有哪些证据？
3. 地中海饮食对心血管风险有什么证据？
4. 限钠饮食对高血压是否真的有帮助？
5. 糖尿病患者为什么需要关注 LDL-C？
```

比较两种回答：

```text
纯学校大模型回答
RAG 系统回答
```

评估维度：

```text
是否有真实引用
引用是否可追溯
是否承认证据不足
是否避免给个人处方建议
结构是否清晰
是否回答了问题核心
```

- 运行完整对比：

```bash
python evaluation.py --mode both
```

- 只评估纯大模型：

```bash
python evaluation.py --mode pure
```

- 只评估 RAG：

```bash
python evaluation.py --mode rag
```

- 控制评估数量：

```bash
python evaluation.py --mode both --limit 2
```

- 输出结果：

```text
evaluation_outputs/evaluation_时间戳.json
evaluation_outputs/evaluation_时间戳.md
```

### 验收标准

- 形成一张对比表。
- 能说明 RAG 相比纯大模型的优势和局限。
- Markdown 报告中包含汇总表、逐题回答和启发式指标。

## 演示时可以强调的亮点

- 系统不是直接让大模型凭空回答，而是先检索真实医学证据。
- 回答中每个关键结论都要求带引用。
- 可以在证据不足时拒绝过度推断。
- 支持学校本地化大模型 API，数据请求默认在校内模型服务中处理。
- 本地向量检索可使用 MacBook MPS 加速。

## 当前不建议优先做的事

- 不建议一开始就做复杂 Agent。
- 不建议先做大量前端美化。
- 不建议接入太多数据源，先把 PubMed 和 ClinicalTrials.gov 做扎实。
- 不建议追求医疗级临床结论，本项目定位仍然是课程 MVP 和技术演示。
